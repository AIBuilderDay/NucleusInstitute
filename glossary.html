The big picture

Nucleus connects two sides of Utah's deep-tech ecosystem.

Talent

People who want to plug into a startup — execs, operators, students, interns, advisors, mentors, board members, investors, service providers. Each one has a profile describing what they want, what they bring, and what they expect (comp, location, risk).

Startups

Utah companies (life sciences, AI, defense/aerospace, cyber, energy, advanced manufacturing, fintech, software). Each has a profile describing their stage, who they need, what they pay, what skills they require.

The matcher

Given a person, rank the startups that fit best (and vice versa). Phase 1 uses pure rules — no AI yet. Future matchers will add embeddings and an agentic Claude-driven matcher, but they'll all return the same shape so the UI doesn't change.

The 5 Nucleus Networks

When someone signs up at nucleusutah.org/contact, they pick one of these five named networks. It's the user's self-declared bucket — separate from role category, which is finer-grained.

Operator operator

Execs and senior operators looking for full-time or fractional roles at a startup. Think: a VP Sales who wants to be a fractional CRO at three pre-seed companies.

Mentor mentor

Informal, time-flexible. Volunteers who'll grab coffee with a founder once a month. No pay, no equity expected.

SME Advisor sme_advisor

Formal subject-matter expert on an advisory board. Specialized in AI / energy / life sciences / aerospace / defense / cyber / advanced manufacturing. Often paid in equity.

Venture venture

Angel investors and active VCs focused on Utah deep tech. Matched by check size, stage, sector, lead vs. follow.

Service Provider service_provider

Creative / legal / operational / technical service firms supporting startups. Often offer startup-friendly terms (deferred billing, equity for services).

Role categories

Finer than networks. role_category is what drives which weights and which hard filters apply during matching. A mentor and an executive get scored very differently even if they pick the same sectors.

How matching works (Phase 1: RuleFilter)

Two stages. First we drop impossible matches outright (hard filters). Then, for the survivors, we score on seven dimensions and weight them by role category.

Stage A: Hard filters → pass / fail (with a "blocker" message if failed) Stage B: Soft scoring → for each survivor, weighted sum of 0.0–1.0 dimension scores Final: score 0.0–1.0 + reasons[] (why it matched) + blockers[] (why it didn't)

Every dimension that scores ≥ 0.5 emits a human-readable reason ("Strong sector overlap: AI"). Those become the bullet points on the match card. Hard-filter failures emit blockers ("Salary gap: $150,000 > $140,000").

Hard filters (the eliminators)

If any of these fail, the match is removed from the candidate list. No partial credit.

Hard Availability

The talent's availability (full_time / part_time / fractional / advisory / internship) must be in the startup's availability_open_to list.

Example blocker: "Availability mismatch: talent wants full_time, startup needs ['fractional']"

Hard Role category

The talent's role category (executive / mentor / student / etc.) must be in the startup's role_categories_open_to list. A startup that only wants advisors won't see executives in their results.

Hard Compensation type

Skipped entirely for mentors. Otherwise, the talent's comp_expectation_type (salary / equity / salary+equity) must be compatible with what the startup offers.

Example: talent wants salary, startup only offers equity → blocked.

Hard Minimum salary

Only applies when the talent expects salary or salary+equity. The startup's comp_max_salary_usd must be at least the talent's comp_min_salary_usd.

Example blocker: "Salary gap: $150,000 > $140,000" (talent's floor exceeds startup's ceiling).

Hard Location

Same state, OR either side has remote_ok = true. So a Salt Lake City startup that's open to remote will still see a Provo candidate; a non-remote startup in California won't see a Utah-only candidate.

The 7 scoring dimensions

Survivors of the hard filters get scored on these. Each yields 0.0–1.0. The final score is the weighted sum (weights depend on role category — see below).

Default weights (executives, operators, investors, service providers)

DimensionWeightWhat it measures
role0.25Title overlap — talent wants CTO, startup needs CTO.
sector0.20Sector overlap — talent likes life_sciences, startup is in life_sciences.
skills0.20How many required skills the talent covers (+ half-credit for nice-to-haves).
stage0.10Talent wants seed-stage companies, startup is seed.
mission0.10Mission keyword overlap — both list "climate" or "health".
location0.10Same city > same state > remote-compatible.
risk0.05Talent's risk tolerance vs. startup's stage. Earlier stage = higher risk.

Per-dimension detail

Dimension Role match

Inputs: talent.role_titles_seeking (list) ∩ startup.roles_needed (list).

Score: overlap-ratio. If at least one title matches, you get partial credit; full overlap = 1.0.

Reason emitted: "Role overlap: cto, fractional_exec"

Dimension Sector match

Inputs: talent.sectors_of_interest vs. startup.sector (primary) + startup.sectors_secondary.

Score: 1.0 if you hit the startup's primary sector, 0.6 if only secondary, 0.0 otherwise.

Reason emitted: "Sector overlap: ai"

Dimension Stage match

Inputs: talent.stage_preference (list) vs. startup.stage (single).

Score: binary. 1.0 if the startup's stage is in the talent's preferred set, 0.0 otherwise. No partial credit for adjacent stages.

Reason emitted: "Stage match: pre_seed" — meaning the talent wanted pre_seed, and this startup is pre_seed.

Dimension Skill match

Inputs: talent.skills vs. startup.required_skills + startup.nice_to_have_skills.

Score: (matched required / total required) + half-credit for nice-to-haves, capped at 1.0. Required dominates.

Reason emitted: "Skill overlap: python, fda regulatory" (only emitted when score ≥ 0.5).

Dimension Mission match

Inputs: talent.mission_keywords ∩ startup.mission_keywords (case-insensitive).

Score: Jaccard-ish — overlap size relative to combined set.

Reason emitted: "Mission alignment: climate, health"

Dimension Location match

Inputs: city, state, remote_ok flags on both sides.

Score: 1.0 same city · 0.7 same state · 0.4 remote-compatible · 0.0 otherwise.

Dimension Risk alignment

Inputs: talent.risk_tolerance (low/medium/high) vs. a stage-derived expected risk:

idea / pre_seed → high · seed / series_a → medium · growth → low

Score: 1.0 exact match · 0.5 one bucket off · 0.0 two buckets off.

Why this matters: a "low risk" CFO who wants stable equity probably shouldn't be matched to an idea-stage napkin sketch.

Anatomy of a match result

Every matcher (rule, embedding, agentic) returns the same MatchResult shape so the frontend match card renders identically.

{ "startup_id": "...", // or talent_id, depending on direction "score": 0.78, // final 0.0 - 1.0 weighted sum "passed_hard_filters": true, "dimension_scores": { "role": 1.00, // talent wanted CTO, startup needs CTO "sector": 1.00, // both life_sciences "stage": 1.00, // talent wanted seed, startup is seed "skills": 0.66, // 2 of 3 required skills "mission": 0.50, // some keyword overlap "location": 0.70, // same state, different city "risk": 1.00 // medium risk vs. seed stage = match }, "reasons": [ // dimensions that scored ≥ 0.5 "Role overlap: cto", "Sector overlap: life_sciences", "Stage match: seed", "Skill overlap: python, fda_regulatory" ], "blockers": [] // would contain hard-filter failures }

Score

Final 0.0–1.0 number. Weighted sum of dimension_scores using the role-category weights. Same scale across all matchers.

passed_hard_filters

Boolean. If false, the match is shown to the user as "blocked" with a list of blockers. We still compute and return dimension scores even on blocked matches — the frontend uses them to power a future "you're 80% fit, here's what's missing" gap-analyzer.

dimension_scores

Per-dimension breakdown. Powers the explainable match card UI — each dimension can be a little bar chart.

reasons

Human-readable bullet points. Generated by each dimension that scores ≥ 0.5. These are the "Why matched" lines the user reads.

blockers

Human-readable bullet points generated by hard-filter failures. Things like "Salary gap: $150,000 > $140,000" or "Availability mismatch".
